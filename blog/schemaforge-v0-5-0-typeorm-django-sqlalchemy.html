SchemaForge v0.5.0: TypeORM, Django, and SQLAlchemy — 6 ORM Formats, 36 Direction Pairs

Release SQLAlchemy TypeORM Django v0.5.0

SchemaForge v0.5.0 is our biggest release yet — adding TypeORM (v0.3.0), Django models (v0.4.0), and SQLAlchemy (v0.5.0) bidirectional conversion. SchemaForge now supports 6 ORM formats with 36 direction pairs and 74 passing tests.

Every format can convert to and from every other format through a single command — with zero-loss roundtripping verified by automated tests.

Supported Formats

SchemaForge now supports the full spectrum of ORM schema formats — from raw SQL to every major ORM:

  • SQL DDL — Standard CREATE TABLE statements with types, constraints, enums, and foreign keys
  • Prisma — Prisma schema files (.prisma) with models, enums, and relations
  • Drizzle — TypeScript schema files with pgTable/mysqlTable/sqliteTable factories
  • TypeORM — Entity classes with @Entity, @Column, @PrimaryGeneratedColumn decorators
  • Django — Python models.py classes with CharField, TextField, ForeignKey, and model options
  • SQLAlchemy — Python declarative model classes with Column(), relationship(), and type annotations

New Features

TypeORM Entity Support (v0.3.0)

Parse TypeScript entity classes with full decorator support:

  • @Entity(), @PrimaryGeneratedColumn(), @Column() decorator parsing
  • Type mappings: varchar → string, integer, decimal, boolean, text, json, uuid, timestamp
  • Options parsing: type, length, nullable, unique, default, precision, scale
  • @ManyToOne, @OneToMany, @JoinColumn decorator detection for foreign key mapping

Django Model Support (v0.4.0)

Parse and generate Python Django model classes:

  • Field mappings: CharField, TextField, IntegerField, FloatField, BooleanField, DateTimeField, DateField, TimeField, DecimalField, UUIDField, JSONField, EmailField, FileField, ImageField, URLField, SlugField
  • Options parsing: max_length, null, blank, default, unique, db_index, choices, auto_now, auto_now_add
  • Meta class parsing: db_table, ordering, verbose_name, unique_together
  • ForeignKey and related_name support

SQLAlchemy Model Support (v0.5.0)

Parse and generate Python SQLAlchemy 2.0-style declarative models:

  • Type mappings: String, Integer, Float, Boolean, DateTime, Date, Time, Text, LargeBinary, JSON, Uuid, Numeric, Enum
  • Column options: primary_key, nullable, unique, default, server_default, autoincrement
  • Default value parsing: Python literals, func expressions (func.now(), func.current_timestamp()), text()
  • Index and UniqueConstraint generation via __table_args__
  • Foreign key detection and relationship() skipping

Real Examples

TypeORM → SQL DDL

Parse this TypeORM entity:

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number;

    @Column({ type: "varchar", length: 100 })
    name: string;

    @Column({ type: "varchar", unique: true })
    email: string;

    @Column({ type: "timestamp", default: () => "CURRENT_TIMESTAMP" })
    createdAt: Date;
}

Convert to SQL:

schemaforge convert --from typeorm --to sql --input User.entity.ts

Result:

CREATE TABLE "user" (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255) UNIQUE,
    createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

SQL → SQLAlchemy

Take this SQL DDL:

CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Convert to SQLAlchemy declarative models:

schemaforge convert --from sql --to sqlalchemy --input schema.sql

Result:

from sqlalchemy import Column, Integer, String, Boolean, DateTime, func
from sqlalchemy.orm import declarative_base

Base = declarative_base()

class Users(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True)
    name = Column(String(100), nullable=False)
    email = Column(String(255), unique=True, nullable=False)
    is_active = Column(Boolean, default=True)
    created_at = Column(DateTime, server_default=func.now())

Diff Mode & Batch Mode

In addition to the new format support, v0.5.0 includes two major workflow features:

Schema Diff

Compare two schemas in the same format and see line-level differences — added, removed, and modified tables, columns, indexes, and constraints:

schemaforge diff schema-v1.prisma schema-v2.prisma
schemaforge diff schema.sql schema-updated.sql --format sql

Batch Directory Conversion

Convert entire directories of schema files in a single command:

schemaforge convert --from sql --to prisma --dir ./schemas/
schemaforge convert --from typeorm --to django --dir ./entities/

How It Works

SchemaForge's architecture hasn't changed — but the IR has matured to handle all 6 formats:

  1. Parse — Each format has a dedicated regex-based parser that extracts tables, columns, types, constraints, indexes, and enums into a unified Intermediate Representation (IR)
  2. IR — A lossless intermediate schema representation (tables, columns, enums, indexes with full metadata)
  3. Generate — Each output format has a generator that renders the IR as idiomatic code for that ORM

The result: zero-loss roundtripping. Convert from SQL → SQLAlchemy → SQL and get back the exact schema you started with. This is verified by 74 passing tests covering every conversion direction.
